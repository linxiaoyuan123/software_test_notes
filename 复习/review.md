# 软件测试基础



## 1、软件测试的定义和目的（重点）：

### 1）定义：

- 软件测试是指用**技术手段**验证**软件**是否满足需求。

### 2）目的：

- 发现**软件缺陷**（bug），验证**需求**是否实现，评估**软件质量**（功能/性能安全）。



## 2、软件测试的原则：

1）**测试证明软件存在缺陷**：测试的目的是为了**发现错误**，而不是证明软件没有错误。

2）**穷尽测试是不可能的**：具体的测试操作不可能将**所有情况**逻列出来，所以需要采用**测试用例设计方法**来做**最有可能**发现错误的测试用例。

3）**测试应尽早介入**：测试不应该只在开发完成后才开始，**需求评审**、**设计评审阶段**测试人员就应该介入，以便尽早发现和预防缺陷。

4）**缺陷的集群性**：大部分的缺陷往往集中在少数模块中。如果某个模块发现的Bug特别多，那么它很可能还有更多未发现的Bug。

5）**杀虫剂悖论**：重复使用**相同的测试用例**，会发现的新Bug越来越少。测试用例需要定期评审和更新，并加入新的测试场景。

6）**测试活动依赖于测试背景**：不同的软件（如银行系统、手机游戏、电商网站）需要采用不同的测试策略、侧重点和强度。



## 3、软件测试分类：

### 1）按测试阶段（重点，测试级别）

| 测试级别             | 测试对象               | 主要参与者      | 目的与重点                                                   |
| -------------------- | ---------------------- | --------------- | ------------------------------------------------------------ |
| 单元测试             | 单个函数、类、模块     | 开发人员        | 验证代码的最小设计单元是否按设计工作，关注内部逻辑           |
| 集成测试（接口测试） | 模块间的接口与交互     | 开发/测试工程师 | 暴露模块间接口定义、数据传输、交互中的缺陷                   |
| 系统测试             | 完整的、集成的软件系统 | 测试工程师      | 在模拟真实环境下的整体验证，包括：功能测试、性能测试、安全测试、兼容性测试等 |
| 验收测试             | 验收测试               | 最终用户/客户   | 确认系统是否满足用户业务需求，是否能够交付，是交付前的最后一道关卡 |

### 2）按测试对象是否可见划分（测试方法）

| 测试方法               | 源代码可见性 | UI功能可见性 | 谁来测试   |
| ---------------------- | ------------ | ------------ | ---------- |
| 黑盒测试（功能测试用） | 不可见       | 可见         | 测试工程师 |
| 白盒测试               | 可见         | 不可见       | 开发人员   |
| 灰盒测试（集成测试用） | 部分可见     | 不可见       | 测试工程师 |

### 3）按测试目的划分（测试类型）

#### a）功能测试：

- 验证软件的功能是否满足需求

#### b）非功能测试：

- 性能测试：包括**负载测试**、**压力测试**、**并发测试**等。考察系统在不同负载下的表现。
- 兼容性测试：在**不同浏览器**、**操作系统**、**设备**上测试软件。
- 安全性测试：发现系统潜在的**安全漏洞**。
- 易用性测试：检验软件是否**好用**。
